  • You can't avoid Side Effects. Not all functions are pure. Not everything has to be immutable. Functional style is about using less brain power and making things simpler. Functional programming is about small, composable things that fit together
  • OOP thinks in terms of a large system. What your objects are and how that your objects interact and the bigger picture.
    Functional style involves thinking about things at the smallest level possible
    The reality is you need to consider both, that's why functional style works in JS
  • I don't want to think about more than 1 thing at any given time. functional style helpers. array.map
  • Pure Function A
    The function always evaluates the same result value given the same argument values
    math.random() // Not Pure

    function add(a, b){return a+b} // Pure
  • Pure Function B
    Evaluation of teh result does not cause any semantically observable side effect or output, such as mutation of mutable objects or output to I/O devices

    let sum = 0;

    function add(a) {sum = sum + a} // Not Pure     that requires too much brainpower because stuff ie: sum is another unnecessary thing to pay attention to.

  • Currying
    in mathmatics and computer science, currying is the technique of translating the evaluation of a function that takes multiple arguments (or a tuple of arguments) into evaluating a sequence of functions, each with a single argument.
    let sum = add(a,b)
    becomes let sum = add(a)(b); Currying Version
